# Claude Code 技术入门详解

> 面向完全没有技术背景的读者，用最通俗的语言解释 Claude Code 是如何工作的

---

## 先说个故事：Claude Code 是什么？

想象一下，Claude Code 是一个**超级能干的 AI 助手**，你可以对它说话，让它帮你写代码、改 bug、管理项目。

它不是那种只会回答问题的聊天机器人，而是一个**会真正动手干活**的助手：
- 你说"帮我修复登录 bug"，它真的会去改代码
- 你说"帮我重构这个模块"，它真的会去写代码并提交
- 你说"帮我研究这个新框架"，它真的会去搜资料、分析代码

下面我们一步一步解释它是怎么做到的。

---

## 第一部分：核心概念入门

### 1.1 什么是 CLI？

**CLI = Command Line Interface（命令行界面）**

就是那种黑屏幕白字的界面，需要输入命令来操作电脑。Windows 上的"命令提示符"、Mac 上的"终端"都是 CLI。

Claude Code 就是这样一个命令行程序，你可以在终端里运行它：

```bash
# 启动 Claude Code
claude

# 或者直接给它一个任务
claude "帮我写一个计算器程序"
```

### 1.2 什么是 AI Model（AI 模型）？

**AI 模型就像一个超级聪明的"大脑"**

你可以把它想象成一个：
- 读过几乎整个互联网的图书管理员
- 会写代码的图书管理员
- 理解你说话意思的图书管理员

Claude Code 底层使用的是 Claude 模型，这个模型由 Anthropic 公司训练。

### 1.3 什么是 Token（令牌）？

**Token = 文字的"碎片"**

当你输入一段文字让 AI 处理时，系统会把文字切成一个个小块，每块叫一个 token。大致规则：
- 英文：1个单词 ≈ 1-2个 tokens
- 中文：1个汉字 ≈ 1个 token

**为什么 token 很重要？**

因为 AI 模型一次能处理的 token 数量是有限的（这叫"上下文窗口"）。就像一根绳子有长度限制一样，AI 处理信息的能力也有限制。

## 第二部分：Claude Code 是如何工作的？

### 2.1 整体工作流程

当你对 Claude Code 说"帮我写一个网页"时，发生了什么：

```
第1步：你输入"帮我写一个网页"
         ↓
第2步：Claude Code 把你的话转成文字（prompt）
         ↓
第3步：把文字发送给 Claude AI 模型
         ↓
第4步：AI 模型思考并决定要做什么
         ↓
第5步：AI 可能需要调用工具（写文件、运行命令等）
         ↓
第6步：把工具执行结果返回给 AI
         ↓
第7步：AI 继续思考或直接回答你
```

这个过程可能会重复很多次，就像人和人对话一样，一问一答，直到任务完成。

### 2.2 什么是 Prompt（提示词）？

**Prompt = 给 AI 的"说明书"**

当你对 AI 说"帮我写代码"，AI 并不知道：
- 你的项目在哪里？
- 用什么编程语言？
- 要遵守什么规范？

所以 Claude Code 会生成一份详细的 prompt，包含：
- 你的项目信息（项目结构、使用的技术）
- 可用工具（能做什么：读文件、写文件、执行命令等）
- 规则和要求（代码规范、提交规范等）
- 当前的对话内容

这份 prompt 就是 AI 的"上下文"，让它能理解任务并正确执行。

## 第三部分：Agent 系统（最核心的概念）

### 3.1 什么是 Agent？

**Agent = AI 派出去干活的"分身"**

你可以把 Agent 想象成：
- 公司的**外包团队**
- 你派出去的**小助手**
- 会自主工作的**机器人**

Claude Code 可以派出去多个"分身"（Agents）同时工作，每个分身都有自己的任务。

### 3.2 Coordinator Mode（协调者模式）

**这是 Claude Code 最强大的功能之一**

当你启用这个模式时，Claude Code 变成一个**项目经理**，而不是亲自干活的员工。

```
普通模式：
┌─────────────────────────────────┐
│  你 → Claude Code（自己干活）    │
└─────────────────────────────────┘

Coordinator 模式：
┌─────────────────────────────────────────┐
│                                          │
│  你 → Claude Code（项目经理）            │
│         ↓ 分配任务                       │
│    ┌────┼────┐                          │
│    ↓    ↓    ↓                          │
│  Worker1 Worker2 Worker3  ← 外包团队     │
│  （并行工作）                            │
└─────────────────────────────────────────┘
```

**为什么这样设计？**

因为：
1. **并行工作快**：一个任务拆成10份，同时做，当然比一个一个做快
2. **隔离性好**：每个 Worker 独立工作，不会互相干扰
3. **更专业**：不同 Worker 可以做不同类型的工作

### 3.3 Worker Agents（工人代理）

**Worker = 具体执行任务的人**

Worker 可以做：
- 研究代码（搜索文件、理解逻辑）
- 写代码（实现功能）
- 验证代码（运行测试）
- 提交代码（git 操作）

每个 Worker 都有：
- **明确的工具清单**（能做什么）
- **具体的任务描述**（要什么结果）
- **独立的上下文**（只记得自己需要的信息）

### 3.4 一个具体的例子

**任务**：给整个项目添加 TypeScript 类型注解

**普通方式**：
```
你 → Claude Code → 一个一个文件改 → 慢
```

**Coordinator 方式**：
```
你：给所有文件添加 TypeScript 类型注解

Claude Code（项目经理）：
  好的！让我先分析一下项目结构...
  
  阶段1：研究（并行3个Worker）
  ├─ Worker A：研究 src/api 目录
  ├─ Worker B：研究 src/components 目录  
  └─ Worker C：研究 src/utils 目录
  
  阶段2：综合
  理解所有发现，制定计划
  确定：共50个文件需要改，分成5组
  
  阶段3：实施（并行5个Worker）
  ├─ Worker 1：改第1-10个文件
  ├─ Worker 2：改第11-20个文件
  ├─ Worker 3：改第21-30个文件
  ├─ Worker 4：改第31-40个文件
  └─ Worker 5：改第41-50个文件
  
  阶段4：验证
  ├─ 验证 Worker 1 的成果
  └─ 验证 Worker 2 的成果
  ...

结果：原本需要2周的工作，2小时完成
```

### 3.5 Workers 之间怎么交流？

**用 XML 格式的报告**

Workers 完成工作后，会发一个"任务通知"：

```xml
<task-notification>
  <task-id>agent-123</task-id>
  <status>completed</status>
  <summary>完成了 API 模块的类型注解添加</summary>
  <result>添加了 23 个文件的类型定义，修复了 5 个类型错误</result>
  <usage>
    <total_tokens>15000</total_tokens>
    <tool_uses>45</tool_uses>
    <duration_ms>30000</duration_ms>
  </usage>
</task-notification>
```

这种格式让项目经理（Claude Code）容易解析和处理。

### 3.6 什么时候派新 Worker vs 继续用旧的？

这是协调者要做的重要决策：

| 情况 | 应该怎么做 |
|------|-----------|
| 研究完正好要改那些文件 | **继续用旧的**（它有上下文） |
| 研究了很多但只改一小部分 | **派新的**（干净的上下文） |
| 上次尝试失败了 | **继续用旧的**（它记得错误） |
| 要验证别人刚写的代码 | **派新的**（独立视角） |

## 第四部分：Tools 系统（AI 能做什么）

### 4.1 什么是 Tool？

**Tool = AI 的"手和脚"**

AI 模型只会"思考"（处理文字），但要真正干活，需要工具。

Claude Code 给 AI 配备了很多工具：

| 工具名称 | 功能 | 相当于 |
|---------|------|--------|
| Bash | 执行命令 | 在终端输入命令 |
| Read | 读文件 | 打开文件看内容 |
| Edit | 改文件 | 修改文件内容 |
| Write | 写文件 | 创建新文件 |
| Grep | 搜索内容 | 在文件里找文字 |
| Glob | 找文件 | 按名字找文件 |
| Agent | 派分身 | 启动新 Worker |

### 4.2 工具怎么工作的？

```
AI：我要读取 config.json 文件
        ↓
系统：好的，让我执行 Read 工具
        ↓
工具：读取文件内容返回
        ↓
系统：把结果发给 AI
        ↓
AI：收到！现在我知道了配置内容
```

### 4.3 权限系统

**为什么需要权限？**

因为 AI 操作的是你真实的项目文件！需要确认你同意这些操作。

```
┌────────────────────────────────────────┐
│  AI 想要：删除 src/old 文件夹            │
│                                          │
│  系统弹出提示：                          │
│  "AI 想要执行删除操作，同意吗？"         │
│                                          │
│  [允许一次] [总是允许] [拒绝]             │
└────────────────────────────────────────┘
```

**权限模式**：

| 模式 | 说明 |
|------|------|
| `default` | 默认，每次操作都问 |
| `plan` | 规划模式，只问重要操作 |
| `acceptEdits` | 自动接受所有编辑 |
| `bypassPermissions` | 跳过所有确认（危险） |

## 第五部分：Skills 系统（可扩展的能力）

### 5.1 什么是 Skill？

**Skill = 预设好的工作流程模板**

想象一下：
- Skill 就像餐厅的"菜单"
- 你不用告诉厨师怎么做菜，只需点菜
- 厨师知道标准做法

例如 `/verify` 这个 skill：
- 你：执行验证
- 系统：知道要用什么工具、什么步骤来验证

### 5.2 Skill 的格式

Skills 用 Markdown 文件编写，包含两部分：

**1. 配置（YAML frontmatter）**：
```yaml
---
name: verify                           # 名字
description: 验证代码变更是否正确     # 描述
when_to_use: 当你需要验证代码时使用   # 什么时候用
effort: medium                       # 投入程度
---
```

**2. 内容（具体步骤）**：
```markdown
# 验证技能

## 执行步骤

1. 运行单元测试
   - 找到 package.json
   - 执行 npm test

2. 运行集成测试
   - 启动服务
   - 测试关键功能

3. 检查代码覆盖率
   - 确保超过 80%
```

### 5.3 几个重要的内置 Skills

#### `/batch` - 批量处理大师

这是最复杂的 skill，能把大任务分解成很多小任务，然后**并行**执行。

例如："给所有组件加上 TypeScript"

```bash
/batch 给所有组件加上 TypeScript 类型注解
```

它会：
1. 先研究项目结构
2. 分解成几十个小任务
3. 启动几十个 Workers **同时**工作
4. 跟踪每个 Worker 的进度
5. 收集结果汇报给你

#### `/verify` - 验证专家

帮你验证代码变更是否正确工作：
- 运行测试
- 检查边界情况
- 生成验证报告

### 5.4 Skill 从哪里来？

```
加载顺序（后面的覆盖前面的）：
1. 系统内置 skills（官方提供的）
2. Policy skills（公司/团队规定的）
3. 用户 skills（你自己创建的）
4. 项目 skills（项目目录下的）
```

### 5.5 什么是 Commands（命令）？

**Command = 内置的快捷功能**

想象一下：
- Command 就像电视遥控器上的**快捷按钮**
- 你按一下，就能直接执行某个操作
- 不需要说话告诉 AI 要做什么

**Commands 和 Skills 的区别**：

| 特性 | Skill | Command |
|------|-------|---------|
| 定义方式 | Markdown 文件 | 代码实现 |
| 复杂度 | 可以很复杂 | 通常简单直接 |
| 用途 | 完整工作流程 | 单一功能 |
| 例子 | `/batch` 批量处理 | `/help` 显示帮助 |

### 5.6 常用 Commands 一览

```
基础命令：
  /help     → 显示所有命令的帮助
  /exit     → 退出 Claude Code
  /clear    → 清屏（清除当前显示）
  /compact  → 压缩对话历史（释放空间）
  /plan     → 开启规划模式

配置命令：
  /config   → 打开配置面板
  /theme    → 更改主题颜色
  /model    → 切换使用的 AI 模型

代码管理：
  /commit   → 提交代码更改
  /review   → 代码审查
  /diff     → 查看当前更改

工具命令：
  /doctor   → 检查问题（帮你诊断）
  /usage    → 查看使用了多少
  /feedback → 发送反馈意见
```

### 5.7 如何使用 Commands？

**方法一：直接输入**

```
在提示框输入：/help
然后按回车
```

**方法二：通过 Skill 调用**

AI 模型可以在需要时调用 Commands：
- AI 说："我将使用 `/verify` 来验证这个变更"
- 系统自动执行 verify 命令

### 5.8 命令的执行方式

Commands 有三种执行方式：

```
1. local（本地执行）
   - 直接运行，结果显示在屏幕上
   - 例如：/help、/clear、/compact

2. local-jsx（渲染界面）
   - 打开一个交互式界面
   - 例如：/config 打开配置面板

3. prompt（送入 AI）
   - 把命令内容变成 prompt，让 AI 处理
   - 例如：/batch 触发复杂的并行处理
```

## 第六部分：上下文管理（Compaction）

### 6.1 什么是上下文窗口限制？

就像手机的内存有限，AI 模型处理信息的能力也有限。

假设 Claude 模型最多能处理 200,000 个 tokens，当对话很长时：

```
┌─────────────────────────────────────┐
│  对话历史 190,000 tokens            │
│  ┌─────────────────────────────┐    │
│  │ 第一天的对话...             │    │
│  │ 第二天的对话...             │    │
│  │ 第三天的对话...             │    │
│  │ 今天新消息...              │    │
│  └─────────────────────────────┘    │
│                         200,000 💥  │
│                        满了！        │
└─────────────────────────────────────┘
```

### 6.2 什么是 Compaction？

**Compaction = 压缩对话历史**

当对话太长时，Claude Code 会：
1. 把**旧对话**压缩成**一段摘要**
2. 保留最近的重要上下文
3. 这样就能继续对话了

```
压缩前：
┌─────────────────────────────────────┐
│  对话1：我想要登录功能...             │
│  对话2：好的，用 JWT...             │
│  对话3：再加个注册...               │
│  对话4：注册要邮箱验证...           │
│  ...（100条对话）                   │
└─────────────────────────────────────┘

压缩后：
┌─────────────────────────────────────┐
│  [摘要] 用户要求实现登录和注册功能， │
│  包括 JWT 认证和邮箱验证...          │
│  ───────────────────────────────     │
│  最近3条对话...                    │
└─────────────────────────────────────┘
```

### 6.3 压缩是自动的吗？

是的！Claude Code 会自动检测什么时候需要压缩：
- 当 token 数量接近限制时
- 当历史太长时

也可以手动触发：
```
/compact   ← 手动压缩对话
```

## 第七部分：Plan Mode（规划模式）

### 7.1 什么是 Plan Mode？

**Plan Mode = AI 先想清楚再动手**

就像建筑要先画图纸，写代码也要先做规划。

```
普通模式：
你：给我加个支付功能
AI：好的（直接开始写）→ 可能改了很多地方才发现问题

Plan Mode：
你：给我加个支付功能
AI：让我先研究一下，然后给你一个计划
      ↓
     你审查计划
      ↓
     你说：可以，开始吧
AI：好的（按计划执行）
```

### 7.2 Plan Mode 怎么做？

```
阶段1：研究
├─ 了解现有代码结构
├─ 理解相关模块
└─ 找出最佳实现方式

阶段2：规划
├─ 列出具体步骤
├─ 标出关键文件
└─ 预估风险

阶段3：你的审批
└─ 你同意计划

阶段4：执行
└─ AI 按计划工作
```

### 7.3 Plan Mode 的优势

1. **避免浪费**：先想清楚再写，避免写了又改
2. **你掌控**：你可以调整计划
3. **更安全**：重要决策经过深思熟虑

## 第八部分：技术实现细节

### 8.1 技术栈

| 技术 | 作用 |
|------|------|
| **Bun** | 运行时的 JavaScript 引擎（比 Node 快） |
| **TypeScript** | 带类型的 JavaScript（更可靠） |
| **React** | 构建 UI 的框架（Claude Code 的界面用它） |
| **Ink** | 专为终端设计的 React（渲染界面） |
| **Zod** | 数据验证库（确保数据格式正确） |

### 8.2 为什么用 Bun 而不是 Node.js？

Bun 比 Node.js 快很多，对于 Claude Code 这种需要频繁调用工具的程序很重要。

### 8.3 什么是 Monorepo？

Claude Code 的代码组织方式：
```
claude-code/
├── packages/           # 内部包
│   ├── @ant/         # Chrome 扩展相关
│   └── ...           # 其他内部模块
└── src/               # 主要源码
    ├── tools/        # 工具实现
    ├── skills/       # Skill 系统
    ├── services/     # 服务（API、压缩等）
    └── ...
```

所有包在一个大项目里，但各自独立。

### 8.4 Feature Flags（功能开关）

Claude Code 有很多实验性功能，用"开关"控制：

```typescript
// 如果开关打开，就启用这个功能
if (feature('EXPERIMENTAL_FEATURE')) {
  // 新功能代码
}
```

在反编译版本中，所有开关都关闭了（功能被禁用）。

### 9.2 核心文件作用

## 第十部分：常见问题

### Q1: Claude Code 和 ChatGPT 有什么区别？

| | ChatGPT | Claude Code |
|--|---------|------------|
| 界面 | 网页/App | 命令行程序 |
| 工作方式 | 一问一答 | 可以调用工具、读写文件 |
| 项目理解 | 每次需要粘贴代码 | 直接读取你的项目 |
| 自动化 | 弱 | 强（可以派多个 Agent） |

### Q2: Claude Code 可以替代 IDE 吗？

不完全是。Claude Code 擅长：
- 快速实现新功能
- 重构和批量修改
- 研究和分析代码

但 IDE（如 VS Code）更适合：
- 实时编辑
- 调试
- 长时间的开发工作

### Q3: Coordinator Mode 适合什么场景？

**适合**：
- 大规模重构（几十个文件）
- 研究+实施的复杂任务
- 需要并行的批量工作

**不适合**：
- 小任务（简单问答、单个文件修改）
- 需要频繁交互确认的任务

### Q4: 使用 Claude Code 安全吗？

- Claude Code 只会操作你授权的目录
- 所有危险操作（删除、覆盖）会请求确认
- 不会自动提交代码（除非你要求）
- 可以设置只读模式（禁止写文件）

## 总结：Claude Code 的架构哲学

### 核心理念

1. **AI 不是万能的，但 AI + 工具 = 强大**
   - 单独的 AI 只能说话
   - 配上工具就能真正干活

2. **并行 > 串行**
   - 一个 AI 改 100 个文件要很久
   - 10 个 AI 同时改就快 10 倍

3. **协调 > 事必躬亲**
   - 好的项目经理不需要自己做
   - 派任务、跟踪结果、汇总汇报

4. **自动管理复杂细节**
   - 你不需要管 token 限制
   - 你不需要管什么时候压缩历史
   - 系统自动处理

5. **渐进式复杂度**
   - 从简单开始
   - 需要时解锁高级功能
   - 不会让新手不知所措

### Claude Code 的本质

```
Claude Code = 一个超级能干的 AI 助手
           + 完整的工具箱（读、写、搜索、执行）
           + 多 Agent 编排系统（并行工作）
           + 自动上下文管理（不怕长对话）
           + 可扩展的 Skill 系统（自定义工作流）
```

它让一个人可以完成原本需要一个团队才能完成的工作。
